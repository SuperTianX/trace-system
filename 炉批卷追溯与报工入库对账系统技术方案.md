## 1. 总体架构设计

### 1.1 架构风格

采用**单体架构 + 模块化分包**，按业务领域分包管理，兼顾开发效率与后期可维护性。单一 Spring Boot 应用，内聚所有业务逻辑，通过包结构划分领域边界。

### 1.2 分层架构

```
┌──────────────────────────────────────────┐
│           Controller 层                   │  ← REST API 入口，参数校验
│   heat / slab / rollBatch / coil / ...    │
├──────────────────────────────────────────┤
│           Service 层                      │  ← 业务逻辑编排，事务管理
│   heat / slab / rollBatch / coil / ...    │
├──────────────────────────────────────────┤
│           Domain / Model 层               │  ← 领域实体、值对象、枚举
├──────────────────────────────────────────┤
│           Mapper 层 (DAO)                 │  ← MyBatis-Plus 数据访问
├──────────────────────────────────────────┤
│           Config / Common 层              │  ← 全局配置、工具类、异常处理
└──────────────────────────────────────────┘
```

### 1.3 系统架构图

```
┌───────────┐  ┌──────────────┐  ┌──────────────┐
│  Vue 3    │  │ 第三方系统    │  │  外部工具     │
│  Element  │  │ MES / ERP    │  │  Nacos? 否   │
│  Plus     │  │ (接口对接)    │  │  直连即可    │
└─────┬─────┘  └──────┬───────┘  └──────────────┘
      │               │
┌─────┴───────────────┴──────────────────────────┐
│           Spring Boot 3.2 单体应用              │
│  ┌──────────────────────────────────────────┐  │
│  │          Controller 层                    │  │
│  │  HeatController / SlabController / ...   │  │
│  ├──────────────────────────────────────────┤  │
│  │          Service 层                       │  │
│  │  HeatService / SlabService / ...         │  │
│  │  TraceService / ReconService / ...       │  │
│  ├──────────────────────────────────────────┤  │
│  │          Mapper 层 (MyBatis-Plus)         │  │
│  └──────────────────────────────────────────┘  │
└──────────────────┬─────────────────────────────┘
                   │
┌──────────────────┴─────────────────────────────┐
│              MySQL 8.0 (Docker)                 │
└────────────────────────────────────────────────┘
```

所有服务打包为一个 JAR，通过 Docker 部署，MySQL 也运行在 Docker 中，docker-compose 一键启动。

## 2. 技术选型

| 类别 | 技术 | 版本 | 说明 |
|------|------|------|------|
| 核心框架 | Spring Boot | 3.2.5 | 基础框架 |
| JDK | OpenJDK | 21 | LTS 版本 |
| 数据库 | MySQL | 8.0 | 所有数据存储 |
| ORM | MyBatis-Plus | 3.5.6 | 数据访问层 |
| 认证鉴权 | Spring Security + JWT | 6.x | RBAC 权限控制 |
| API 文档 | Knife4j (Swagger) | 4.x | 接口文档自动生成 |
| 定时任务 | Spring @Scheduled | 内建 | 对账/诊断定时执行 |
| 参数校验 | Jakarta Validation | 内建 | 入参校验 |
| 前端 | Vue 3 + Element Plus | 最新 | 管理后台 |
| 可视化 | ECharts | 5.x | 链路图 / 图表 |
| 构建 | Maven | 3.9+ | 项目构建 |
| 容器化 | Docker + Docker Compose | 最新 | 部署运行 |
| 数据库连接池 | HikariCP | 内建 | 性能最优 |

**关键设计原则：**
- 不引入 ES、MQ、Redis、MinIO 等中间件，降低运维复杂度
- 所有数据存 MySQL，合理设计索引和分页保障性能
- 定时任务使用 Spring 内建 `@Scheduled`，不引入外部调度器
- 部署仅依赖 Docker + Docker Compose，一条命令启动

## 4. 领域模型设计

### 4.1 核心实体关系

```
┌──────────┐       ┌──────────┐       ┌────────────┐
│  炉次    │1───N▶│  铸坯    │1───N▶│  轧制批次   │
│ (Heat)   │       │ (Slab)   │       │ (RollBatch) │
└──────────┘       └──────────┘       └──────┬─────┘
                                             │1
                                             │N
                                        ┌────▼─────┐
                                        │  钢卷     │1──N▶┌──────────┐
                                        │ (Coil)    │─────│质检记录   │
                                        └────┬─────┘     │(QcRecord)│
                                             │            └──────────┘
                        ┌─────────┐          │
                        │报工记录  │◄─────────┤
                        │(WorkRpt)│          │
                        └─────────┘          │
                                        ┌────▼─────┐     ┌──────────┐
                                        │入库/出库  │◄────│库存台账   │
                                        │(Inventory)│     │(Stock)   │
                                        └────┬─────┘     └──────────┘
                                             │
                                        ┌────▼─────┐
                                        │质量异议   │
                                        │(Complaint)│
                                        └──────────┘
```

### 4.2 核心实体简要定义

| 实体 | 业务主键 | 核心字段 | 关联关系 |
|------|----------|----------|----------|
| Heat | heat_id | 冶炼日期、钢种、化学成分、班组、状态 | 1:N → Slab |
| Slab | slab_id | 关联炉号、规格、重量、连铸班组 | N:1 ← Heat, N:1 → RollBatch |
| RollBatch | batch_id | 关联铸坯、产线、工单、工艺参数 | 1:N ← Slab, 1:N → Coil |
| Coil | coil_id | 规格、重量、材质、质量等级、库存状态 | N:1 ← RollBatch |
| QcRecord | 自增id | 关联类型、关联编号、检验项目、判定 | N:1 ← Coil/Heat/Slab |
| WorkReport | 自增id | 工单、工序、批次、卷号、数量、审核 | N:1 ← Coil |
| Inventory | doc_no | 类型(入库/出库)、卷号、数量、仓库 | N:1 ← Coil |
| Stock | coil_id+warehouse | 库存数量、库位、状态 | 1:1 ← Coil |
| ReconDiff | 自增id | 批次、四端数量、差异类型、责任部门 | - |
| ChainBreak | 自增id | 断链类型、关联编号、风险等级 | - |
| Complaint | complaint_id | 客户、卷号、问题、措施、状态 | N:1 ← Coil |

## 5. 数据库设计

### 5.1 数据库与表命名规范

- 数据库名：`trace_db`
- 表前缀：`tr_`
- 字符集：`utf8mb4`，排序规则：`utf8mb4_general_ci`
- 主键统一 `BIGINT AUTO_INCREMENT`，业务主键（如 heat_id, coil_id）建唯一索引
- 所有表包含 `is_deleted`、`create_time`、`update_time` 通用字段

### 5.2 核心表结构

#### 5.2.1 炉次表 (tr_heat)

```sql
CREATE TABLE tr_heat (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    heat_id VARCHAR(50) NOT NULL COMMENT '炉号',
    smelt_date DATE NOT NULL COMMENT '冶炼日期',
    steel_grade VARCHAR(100) COMMENT '钢种',
    c_content DECIMAL(10,4) COMMENT '碳含量(%)',
    si_content DECIMAL(10,4) COMMENT '硅含量(%)',
    mn_content DECIMAL(10,4) COMMENT '锰含量(%)',
    p_content DECIMAL(10,4) COMMENT '磷含量(%)',
    s_content DECIMAL(10,4) COMMENT '硫含量(%)',
    cast_start_time DATETIME COMMENT '开浇时间',
    shift_group VARCHAR(50) COMMENT '冶炼班组',
    equipment_id VARCHAR(50) COMMENT '设备编号',
    status TINYINT NOT NULL DEFAULT 0 COMMENT '0-正常 1-异常',
    abnormal_desc VARCHAR(500) COMMENT '异常说明',
    is_deleted TINYINT DEFAULT 0,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_heat_id (heat_id),
    INDEX idx_smelt_date (smelt_date),
    INDEX idx_steel_grade (steel_grade)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='炉次表';
```

## 7. 核心业务流程设计

### 7.1 正向追溯

```
输入炉号/铸坯号/批次号
    │
    ▼
解析入口节点（查 tr_heat / tr_slab / tr_roll_batch）
    │
    ▼
沿链路正向遍历：
  heat → slab → roll_batch → coil → qc → inventory(in) → inventory(out)
    │
    ▼
并行查询各节点详情（多条 SQL 查各表）
    │
    ▼
逐节点校验状态、完整性 → 标记异常节点
    │
    ▼
组装链路树结构 → 写入 tr_trace_link 快照
    │
    ▼
返回链路数据
```

伪代码示意：

```java
@Service
public class TraceService {

    public TraceLinkVO traceForward(TraceRequest req) {
        // 1. 解析入口
        String entryId = req.getInputValue();
        // 2. BFS 构建链路
        List<TraceNode> nodes = new ArrayList<>();
        if ("heat".equals(req.getInputType())) {
            Heat heat = heatMapper.selectByHeatId(entryId);
            nodes.add(buildHeatNode(heat));
            // 查子节点 slabs
            List<Slab> slabs = slabMapper.selectByHeatId(entryId);
            for (Slab slab : slabs) {
                nodes.add(buildSlabNode(slab));
                // 递归查下游...
            }
        }
        // 3. 校验异常
        for (TraceNode node : nodes) {
            node.setStatus(checkNodeStatus(node));
        }
        // 4. 保存快照
        saveTraceLink(nodes, req);
        // 5. 返回
        return assembleLinkVO(nodes);
    }
}
```

### 7.2 报工入库对账

```
每天凌晨 1:00 定时触发
    │
    ▼
按批次聚合：MES 报工数量、质检合格数量、ERP 入库数量、库存台账数量
    │
    ▼
逐批次比对四端数量（应用容差规则）
    │
    ├── 一致 → 跳过
    └── 不一致 → 写入 tr_recon_diff
    │
    ▼
差异通知 → 责任部门确认处理 → 复核关闭
```

```java
@Service
public class ReconciliationService {

    @Scheduled(cron = "0 0 1 * * ?")  // 每天凌晨1点
    public void executeReconciliation() {
        // 1. 获取各端聚合数据
        List<BatchQty> workReportQtys = workReportMapper.sumByBatch();
        List<BatchQty> qcPassQtys = qcRecordMapper.sumPassByBatch();
        List<BatchQty> erpInboundQtys = inventoryMapper.sumInboundByBatch();
        List<BatchQty> stockQtys = stockMapper.sumByBatch();

        // 2. 取所有批次并集
        Set<String> allBatches = new HashSet<>();
        allBatches.addAll(...);

        // 3. 逐批次比对
        for (String batchId : allBatches) {
            BigDecimal wrQty = findQty(batchId, workReportQtys);
            BigDecimal qcQty = findQty(batchId, qcPassQtys);
            BigDecimal erpQty = findQty(batchId, erpInboundQtys);
            BigDecimal stQty = findQty(batchId, stockQtys);

            if (hasSignificantDiff(wrQty, qcQty, erpQty, stQty)) {
                reconDiffMapper.insert(new ReconDiff(batchId, wrQty, qcQty, erpQty, stQty));
            }
        }
    }
}
```

### 7.3 断链诊断

```
每天凌晨 2:00 定时触发
    │
    ▼
检查以下规则：
  ┌ 批次断链: slab 无 heat, coil 无 batch, 入库无 coil
  ├ 质检缺失: coil 已入库但无质检记录
  ├ 未质检入库: 入库单关联卷无合格质检
  ├ MES/ERP不一致: MES 有报工但 ERP 无入库
  ├ 重复报工: 同批次同工序多条报工
  └ 数量异常: 上下游数量超出容差
    │
    ▼
逐条写入 tr_chain_break → 风险分级(低/中/高/严重)
    │
    ▼
责任分配 → 处理 → 复核关闭
```

## 8. 非功能需求设计

### 8.1 性能保障

| 场景 | 目标 | 措施 |
|------|------|------|
| 页面刷新 | < 2s | SQL 索引优化，分页查询，避免 N+1 |
| 报表查询 | < 5s | 合理聚合查询，数据量千笔以内直接 SQL |
| 单记录操作 | < 3s | 索引命中，事务粒度控制 |
| 数据接口(对接MES/ERP) | < 60s | 接口超时设置，异步处理 |

**数据库优化：**
- 所有查询字段建覆盖索引
- 大批量导入使用 MyBatis-Plus `saveBatch` 批量提交
- 追溯历史链路数据限制保留时间（如 6 个月自动清理）
- 慢查询通过 MySQL 慢查询日志定位优化

### 8.2 安全设计

- **认证**：Spring Security + JWT（access_token 有效期 30 分钟，refresh_token 7 天）
- **鉴权**：RBAC 模型，接口级权限控制，基于角色分配菜单和按钮权限
- **数据校验**：Jakarta Validation 注解统一校验入参
- **防注入**：MyBatis-Plus 参数预编译，禁止 SQL 拼接
- **日志记录**：关键操作（新增、修改、删除、审核、关闭）记录操作日志

### 8.3 可靠性设计

- **事务控制**：Service 层 `@Transactional`，关键业务（如报工→库存联动）保证原子性
- **数据校验**：前端 + 后端双重校验，确保数据完整性
- **幂等处理**：关键接口支持幂等（基于业务主键去重）
- **数据备份**：MySQL 定时 mysqldump 备份
- **日志**：AOP 切面记录操作日志 + 异常日志
